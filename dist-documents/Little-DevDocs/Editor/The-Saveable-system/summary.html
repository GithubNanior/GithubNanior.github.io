Overview

The Saveable system is a core system in ALS that enables certain prefab instances and their references to persist across game sessions.

Related documentations

  • The Registry system

Terminology

  • Saveable object refers to a persistent object that can be spawned into the world, not to be confused with the class SaveableObject.
  • Saveable instance refers to an instance of a Saveable object.
  • Prefab Id or just Id, is an int unique to each Saveable object, each id corresponds to a SaveableObject registered in AssetsManager.saveableRegistry.
  • Instance Id is an int unique per Saveable instance, it's unique across the entire save to allow remembering objects cross-dimension.
  • Instance Code is an int, a tally that increments whenever an Instance Id gets reused, enabling you to distinguish new Saveable instances with reused instance Ids from the previous owner of said Id.
  • Registered SavedObjects refer to SavedObjects that are registered in the SaveableManager, most methods count unregistered SavedObjects as null or invalid.

Multi-form objects

Saveable objects exist in three stages, precision, complexity, whatever you want to call them. The SaveableManager handles conversions between the three.
Note that the lower stage only gets updated when the higher stage is unloaded.

  • Binary: A compact, persistent data format stored on disk. They exist inside their dimension's save file and are thus inaccessible.
    All Saveable instances, regardless of dimension have a binary representation.
  • SavedObject: Parsed from the binary when a dimension is loaded into memory. You're not meant to operate on SavedObjects directly however, but rather by their SaveableReference.
    All Saveable instances in memory have an underlying SavedObject.
  • SaveableObject: Also called the loaded object, this is the visible, interactable GameObject equivalent of a SavedObject.
    Gameplay code can easily load and unload them from their SaveableReference.
